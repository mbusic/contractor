# Deploying to Render

This repo has a `render.yaml` blueprint at the root that provisions three things on Render's
free tier: the Spring Boot backend, the Angular frontend (static site), and a free PostgreSQL
database.

**Why the backend uses Docker:** Render has no native Java runtime (confirmed against
[Render's Native Runtimes docs](https://render.com/docs/native-runtimes) - only Node, Python,
Ruby, Go, Rust, Elixir are supported natively). Render's own guidance for anything else is to
deploy with Docker, so `backend/Dockerfile` is a plain two-stage build (Maven build stage,
`eclipse-temurin` JRE run stage) - Render still builds and hosts it the same way, you just
maintain one Dockerfile instead of zero. This was verified locally with `docker build` and a
real Postgres container before being committed.

## 1. Push this branch and connect the repo

1. Push this repo to GitHub (if not already).
2. In the Render dashboard: **New > Blueprint**, pick this repo. Render reads `render.yaml`
   and shows a preview of the 3 resources it will create (`contractor-db`,
   `contractor-backend`, `contractor-frontend`). Apply it.
3. First deploy will likely fail or come up with broken CORS/photos - that's expected, see
   step 2.

## 2. Fix the cross-service URL placeholders

`render.yaml` wires the database connection automatically, but the backend's
`APP_CORS_ORIGIN` / `APP_BASE_URL` and the frontend's API URL can't be known until Render
assigns the `*.onrender.com` hostnames. After the first deploy:

1. Note the two service URLs Render assigned, e.g.:
   - `https://contractor-backend-xxxx.onrender.com`
   - `https://contractor-frontend-xxxx.onrender.com`
2. In the Render dashboard, on **contractor-backend > Environment**, set:
   - `APP_CORS_ORIGIN` = the frontend URL
   - `APP_BASE_URL` = the backend's own URL
3. Update `frontend/src/environments/environment.prod.ts` locally with the real backend URL
   (`apiUrl` and `fileBaseUrl`), commit, and push - this triggers a frontend rebuild with the
   correct API target baked in (Angular's prod build inlines these values at build time, so
   they can't be set as a runtime env var on Render's static site).

## 3. Env vars set automatically by the blueprint

These come from `render.yaml` and don't need manual entry:

| Var | Source |
|---|---|
| `DATABASE_URL` | Render's `contractor-db` Postgres instance (`connectionString` property - a `postgresql://user:pass@host:port/db` URL, parsed into a JDBC DataSource by `RenderDataSourceConfig.java` since Spring can't consume that URI scheme directly) |
| `APP_JWT_SECRET` | Auto-generated by Render (`generateValue: true`) |
| `PORT` | Injected by Render for every web service, including Docker-runtime ones |

## 4. Env vars you must set manually in the dashboard

| Var | Where | Value |
|---|---|---|
| `APP_CORS_ORIGIN` | contractor-backend | The frontend's `*.onrender.com` URL |
| `APP_BASE_URL` | contractor-backend | The backend's own `*.onrender.com` URL |

These are now filled in in `render.yaml` with this deploy's real URLs
(`contractor-frontend-wgfq.onrender.com` / `contractor-backend-o4uk.onrender.com`) - only
re-check them here if either service ever gets recreated with a different generated hostname.

## Known limitations of this deploy (POC, not fixed by this pass)

- **Uploaded photos do not persist.** Render's free web services have an ephemeral
  filesystem - anything written to `app.upload-dir` (`./uploads`) is lost on every
  redeploy or restart. Fixing this means swapping `FileStorageService` for an object
  storage client (S3/GCS/etc.), which is out of scope here since it's a code change, not
  deployment config.
- **Free Postgres and free web services on Render are not permanent.** Render's free tier
  policies (expiry windows, spin-down after inactivity) change fairly often - check the
  current Render pricing page before relying on this for anything beyond a demo. Expect
  a 30-60s cold start on the backend after it's been idle, and possibly a spun-down static
  site cold path too.
- **No schema migration tool.** `spring.jpa.hibernate.ddl-auto=update` is used for both
  dev and Render profiles since there's no Flyway/Liquibase in the project. Fine for a
  POC; do not use this in a real production setup with existing data.
- **The `render.yaml` field names (`runtime: docker`, `runtime: static`, `fromDatabase`
  properties) were verified against Render's live Blueprint YAML Reference and, for the
  backend, an actual local Docker build + run against a real Postgres container - not just
  guessed. Render does still evolve this spec occasionally, so if a future `render.yaml`
  apply is rejected, re-check https://render.com/docs/blueprint-spec for renamed fields.
